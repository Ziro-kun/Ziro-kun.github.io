# 🔎 프로젝트 개요

- 프로젝트명 : 차세대 자동차관리정보시스템 구축 용역
- 발주처 : 한국교통안전공단(TS)
- 참여 컨소시엄 : 케이사인(주), 한국정보인증(구. 조인트리)
- 프로젝트 기간 : 2022-11-07 ~ 2025-07-06
- 본인 참여 기간: **2024.01 ~ 2025.06 (오픈 전 - 오픈 후 안정화까지 참여)**
- 프로젝트 개요 : 국토교통부 주관 자동차·건설기계 관련 **대규모 공공 서비스의 차세대 시스템을 구축하고 기존 정보의 클라우드 마이그레이션 수행**
    - **1등급 공공 SI 프로젝트**로**,** 당시 자사 역할은 한국정보인증 소속 협력사로써, 기존 공공 서비스를 클라우드 환경으로 전환하면서, 대규모 외부 기관 연계를 안정적으로 수행하기 위한 연계 시스템(EAI/ESB/API Gateway) 구축이 핵심 과제였음

# 🚩 프로젝트 수행과정 요약

- 본 프로젝트는 다수의 공공기관 및 자동차·건설기계 관련 기관과 연계되는 구조로, **REST, SOAP, Socket 통신을 혼합한 메시지 기반 연계 방식 사용**
- 연계 대상 기관은 국내 자동차 관련 업무를 수행하는 공공기관과 자동차 제조사 및 협회 등 민간기관 200 여개, 기관 시스템은 약 **500여 개 이상**, 기관별 시스템 및 인터페이스 수는 약 2,000여 개의 방대한 프로젝트로, 연계 테스트 및 오픈 준비 과정에서 체계적인 관리와 조율이 필수적임
- 본인은 **연계팀 소속**으로, 시스템에 사전 데이터를 입력하고, 기획자가 부재한 환경에서 개발자·API 개발팀·외부 기관 사이의 **확인·전달·조율 역할**을 중심으로 수행하였음
- 연계 테스트 단계에서
    - 인터페이스 정의서 및 테스트 케이스 기반 검증
    - 첨부형 API(기존/신규 병행) 참여 기관 관리
    - 오픈 대상 기관 선별 및 일정 조율
    - 연계 오류 발생 시 로그 분석을 통해 원인 정리 후 개발팀 전달 등의 업무 수행
- 시스템 오픈 이후에는 안정화 단계까지 참여하여 연계 오류 모니터링, 추가 조치사항 정리, 기관 대응 지원을 수행함

# 🪄 활용된 기술 및 환경

### 🔹 시스템/개발 환경

- **Frontend**: Nexacro 기반 행정·업무 시스템
- **Backend**: 공공 SI 표준 환경의 Java/Spring 계열 시스템
- **연계 구조**: EAI / ESB / API Gateway 기반 메시지 연계
- **통신 방식**: REST, SOAP, Socket, FileSystem 등 혼합 구조
- **데이터 흐름**: 메시지 큐 기반 비동기 연계 지원

### 🔹 업무 및 테스트·운영·지원 툴

- **Postman** : REST·SOAP API 테스트 및 검증
- **MobaXterm** : 서버 접속 및 `tail`, `grep` 기반 로그 분석
- **차세대 자동차관리시스템 관리자 포털** : 인터페이스 트랜잭션 상태 확인
- **Microsoft Excel**: 인터페이스 목록, 테스트 진행 현황, 오류 관리
- **PowerPoint**: 테스트 결과 및 오픈 준비 현황 공유 자료 작성
- **한글(HWP)**: 공공 SI 산출물 및 기관 제출 문서 작성·정리

# ⚠️ 프로젝트 과정 중 발생한 문제 및 해결 과정

- **(문제 1) 연계 대상 기관·인터페이스 수가 방대하여 테스트 현황 파악이 어려움**
    
    - 기관별·인터페이스별 테스트 진행 현황을 Excel 기반으로 구조화하여 관리함으로써, 테스트 누락 및 중복 작업을 최소화함
        
    - 로그 확인 및 오류 원인 파악 속도를 높이기 위해 리눅스 기본 명령어와 SQL 조회를 학습·적용하여, 테스트 결과와 로그 정보를 체계적으로 정리·공유함
        
        → 이를 통해 연계 테스트 진행 상황을 한눈에 파악할 수 있는 기반을 마련함
        
- **(문제 2) 연계 오류 발생 시 원인 파악 및 대응에 시간이 소요됨**
    
    - MobaXterm을 활용해 서버 로그를 실시간으로 확인하고, ESB/API Gateway 트랜잭션 흐름과 함께 오류 발생 지점을 정리하여 개발팀에 전달함
        
    - 단순 전달에 그치지 않고, 조치 기한을 명확히 하여 진행 상황을 추적하고, 해결 결과를 요청 기관에 즉시 공유함
        
        → 반복 문의와 불필요한 커뮤니케이션을 줄이고, 참여 기관의 신뢰도 및 운영 안정성을 제고함
        
- **(문제 3) 첨부형 API의 신규·기존 혼합 운영으로 인한 테스트 및 오픈 혼선**
    
    - 오픈 시점 기준 실제 참여 기관만 선별하여 관리하고, 테스트 대상·적용 여부를 명확히 구분하여 공유함
        
    - 레거시 참여기관 정보가 최신화되지 않은 상황에서, 제한된 네트워크 환경을 고려해 개인 장비를 활용하여 연락 채널을 다각화함
        
        → 오픈 대상 기관 범위를 명확히 하여 불필요한 테스트 및 혼선을 최소화함

# 📊 프로젝트 성과 및 기여

- 본인은 회사의 전략기획자로써 대규모 공공 SI 프로젝트에서 개발된 연계 소프트웨어(EAI/ESB/API Gateway)가 **사내 제품으로 자산화되는 과정을 지원함**
    - 기능 구성, 연계 범위, 적용 사례 등을 문서로 정리하여 **회사 제품 저작권 등록을 위한 문서화 작업에 기여**
    - 해당 연계 소프트웨어의 **한국정보통신기술협회 BMT 시험 준비 및 수행을 지원**
        - 시험 대상 기능 범위 정리
        - 관련 문서 정합성 검토
        - 시험 대응 자료 정리 및 내부 공유
- 프로젝트 산출물이 단순 SI 결과물에 그치지 않고 **공공 기준을 충족한 제품으로 검증·자산화되는 전 과정 참여**

# ✅ 회고 및 느낀점

- 개발 코드를 직접 작성하지는 않았지만, 로그 확인 및 개발팀 전달 후 외부 이해관계자에게 설명하는 과정에서 **연계 구조·시스템 흐름·운영 관점**을 깊이 이해할 수 있었음
- 대규모 공공 SI 환경에서는 기술 구현만큼이나 **정리, 문서화, 커뮤니케이션, 조율 능력**이 중요하다는 것을 체감
- 연계 테스트, 제품화, 저작권 등록, TTA BMT까지 이어지는 과정을 경험하며 ‘**일회성 프로젝트가 아닌, 자산으로 남는 시스템’을 만드는 관점**을 갖게 됨
