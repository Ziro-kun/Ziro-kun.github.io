# 깃허브 블로그 자동화 구축 중 - Velog 마이그레이션 준비

요즘 블로그 플랫폼이 정말 많잖아요. Velog도 좋지만, 이번 기회에 **깃허브 블로그로 완전히 이전**하기로 결심했어요. 단순히 옮기는 것만으로는 부족해서 자동화 시스템까지 구축하고 있는 과정을 공유해볼게요.

## 왜 깃허브 블로그로 옮기기로 했나

Velog는 정말 훌륭한 플랫폼이에요. 마크다운 에디터도 쾌적하고, 커뮤니티도 활발하고. 하지만 다음 이유들 때문에 옮기기로 결정했습니다.

- **완전한 소유권**: 자신의 저장소에 모든 글이 남음
- **자동화 가능성**: CI/CD 파이프라인으로 배포 자동화
- **커스터마이징**: 디자인과 기능을 100% 제어 가능
- **버전 관리**: Git으로 글의 모든 변경 이력 추적

## 자동화 시스템의 핵심 구성

현재 구축 중인 자동화 시스템은 크게 세 부분으로 나뉩니다.

### 1. 콘텐츠 마이그레이션 (Content Migration)

Velog의 모든 글을 자동으로 추출해서 마크다운 파일로 변환하는 과정입니다.

- Velog API를 활용한 글 데이터 수집
- 이미지 다운로드 및 로컬 저장
- YAML frontmatter 자동 생성
- 메타데이터(태그, 발행일 등) 정리

### 2. 빌드 및 배포 자동화 (Build & Deploy Automation)

새로운 글을 추가하면 자동으로 배포되도록 설정합니다.

```yaml
# GitHub Actions 예시
name: Deploy Blog
on:
  push:
    branches: [main]
jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Build
        run: npm run build
      - name: Deploy
        run: npm run deploy
```

### 3. 메타데이터 표준화

모든 글이 일관된 형식을 유지하도록 **YAML frontmatter를 표준화**했습니다.

```yaml
---
title: "글 제목"
description: "한 줄 요약"
pubDate: YYYY-MM-DD
tags: ["태그1", "태그2"]
---
```

## 진행 상황과 앞으로의 계획

현재 자동화 시스템의 **기본 골격은 완성**된 상태입니다. 남은 작업은 다음과 같아요.

- 기존 Velog 글 전체 마이그레이션 (약 50개 글)
- 이미지 경로 자동 수정
- 내부 링크 업데이트
- 배포 테스트 및 최적화

## 마치며

블로그 자동화는 처음엔 복잡해 보이지만, 한 번 구축하면 정말 편합니다. 특히 **글 작성에만 집중할 수 있다**는 게 가장 큰 장점이에요.

곧 완전한 마이그레이션이 끝나면, 깃허브 블로그에서 더 자주 뵐 수 있을 것 같습니다. 혹시 비슷한 작업을 고려 중이라면, 자동화 구축 과정을 더 자세히 공유해드릴 수 있으니 궁금한 점 있으면 편하게 물어봐주세요!
